# Chapter 11 — Schema Evolution with Avro

## What this module demonstrates

Kafka topics are append-only logs. Records written today will be read months
from now by consumers that may be running a newer version of the application.
Schema evolution is the discipline of changing message formats without
breaking existing consumers or producers.

Apache Avro solves this with a binary encoding that carries no field names —
only values in a schema-defined order. The reader resolves fields by comparing
the *writer schema* (what produced the bytes) against the *reader schema*
(what the current consumer expects). Avro fills in default values for fields
the writer did not include and silently drops fields the reader does not
recognize.

## The golden rule of backward compatibility

> A new reader schema is **backward compatible** with an old writer schema
> when every field added by the new schema has a **default value**.

`OrderPlacedV2` adds `discountCode` as a nullable union with a `null` default:

```json
{"name": "discountCode", "type": ["null", "string"], "default": null}
```

A V1 consumer reading V2 bytes simply ignores the extra field.
A V2 consumer reading V1 bytes gets `null` for `discountCode` automatically.

## What breaks compatibility

* Adding a field **without** a default value — old consumers cannot fill it in.
* Removing a field that has no default in the reader — the reader has no value
  to use when the field is absent from new writer bytes.
* Changing a field's type to an incompatible type (e.g. `string` → `int`).

```bash
mvn test -pl chapter-11-schema-evolution
```

```bash
# Start Kafka (KRaft mode, no ZooKeeper)
docker compose -f ../chapter-12-observability/docker-compose.yml up -d kafka

mvn spring-boot:run -pl chapter-11-schema-evolution
```

## Moving to production with Schema Registry

The raw `byte[]` approach works for learning but production systems benefit
from Confluent Schema Registry, which enforces compatibility rules at publish
time and stores schemas centrally.

Replace the manual serialization with:

```xml
<dependency>
    <groupId>io.confluent</groupId>
    <artifactId>kafka-avro-serializer</artifactId>
    <version>7.6.0</version>
</dependency>
```

```yaml
spring:
  kafka:
    producer:
      value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer
    consumer:
      value-deserializer: io.confluent.kafka.serializers.KafkaAvroDeserializer
    properties:
      schema.registry.url: http://localhost:8081
      specific.avro.reader: true
```

The serializer prepends a 5-byte magic + schema-ID header to every message,
allowing consumers to fetch the exact writer schema from the registry and
perform correct resolution even when producer and consumer are deployed
independently.
